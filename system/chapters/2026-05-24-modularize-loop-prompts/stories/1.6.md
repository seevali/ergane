## Technical Details

### The 3-Layer Model (for context)

Copied from [the chapter plan](../README.md#recommended-path-hybrid-3-layer-composition) and [the PRD](../prd.md#approach):

Each cached system prompt is built from three concatenated layers:

```
[Layer 1: Execution Context Override]   ← repo-local, stable
[Layer 2: BMAD Persona]                 ← live from .claude/skills/, may be empty
[Layer 3: Demo-Specific Rules]          ← repo-local stack / review rules
```

- **Layer 1** (`scripts/prompts/common/execution-context.md`): Non-interactive overrides that prevent the loop from HALT-ing or greeting users — ensures the loop runs autonomously.
- **Layer 2** (read from `.claude/skills/<skill>/SKILL.md` or fallback to `scripts/prompts/bmad-fallbacks/`): The BMAD agent persona (Scrum Master, Developer, Code Reviewer). This is live-loaded from BMAD, so future BMAD updates flow in automatically.
- **Layer 3** (`scripts/prompts/common/project-conventions.md` + per-role overlays): Stack-specific rules (React, TypeScript, Vite conventions) and role-specific rules (review pass/block criteria, etc.).

Layer ordering is intentional: Layer 1 must come first so its "don't HALT" rules override Layer 2's persona instructions. Layer 3 comes last so stack rules are the most recent instruction.

### Glossary for users forking the demo

If a user forks this demo for a different tech stack (e.g., Vue instead of React, Python instead of Node), they would:

1. Modify `scripts/prompts/common/project-conventions.md` to swap React/Vite/TypeScript rules for their stack's conventions.
2. Create or update `scripts/prompts/<role>/overlay.md` files if they need role-specific customization.
3. They do **not** edit `scripts/ralph-loop.sh` — the loop logic is generic.

This is the key benefit of externalizing prompts: the loop script itself is now portable to other stacks.
