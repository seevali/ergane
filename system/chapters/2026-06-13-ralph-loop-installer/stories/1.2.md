## User Story

As a developer setting up the Ralph Loop installer, I need a functional npm package skeleton with a Commander-based CLI and test infrastructure so that subsequent stories can build on a solid foundation.

## Acceptance Criteria

- `node installer/bin/ralph.js --help` lists all subcommands (`install`, `update`, `uninstall`, `doctor`) and exits with code 0
- `node installer/bin/ralph.js --version` displays the version from `package.json` and exits with code 0
- `cd installer && npm test` passes with a single placeholder test
- `npm run lint` (or syntax check) passes on all JavaScript files

### Step 2: Create `installer/bin/ralph.js`

Create `/home/seevali/projects/Metis/demos/ralph-loop-demo/installer/bin/ralph.js`:

```javascript
#!/usr/bin/env node

import { program } from 'commander';
import { readFileSync } from 'fs';
import { fileURLToPath } from 'url';
import { dirname, join } from 'path';

// Read version from package.json
const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
const packagePath = join(__dirname, '../package.json');
const { version } = JSON.parse(readFileSync(packagePath, 'utf8'));

program
  .version(version, '-v, --version', 'Show version number and exit')
  .helpOption('-h, --help', 'Show help and exit')
  .description('Ralph Loop guided installer — set up a new loop project in minutes');

// Install subcommand (stub)
program
  .command('install')
  .alias('init')
  .description('Install or update a Ralph Loop project')
  .action(() => {
    console.log('install: not yet implemented');
    process.exit(0);
  });

// Update subcommand (stub)
program
  .command('update')
  .description('Update an existing Ralph Loop installation')
  .action(() => {
    console.log('update: not yet implemented');
    process.exit(0);
  });

// Uninstall subcommand (stub)
program
  .command('uninstall')
  .description('Remove a Ralph Loop installation')
  .action(() => {
    console.log('uninstall: not yet implemented');
    process.exit(0);
  });

// Doctor subcommand (stub)
program
  .command('doctor')
  .description('Validate an existing Ralph Loop installation')
  .action(() => {
    console.log('doctor: not yet implemented');
    process.exit(0);
  });

program.parse(process.argv);

// Show help if no args provided (standard CLI convention)
if (process.argv.length === 2) {
  program.outputHelp();
  process.exit(0);
}
```

**Key decisions:**
- Shebang (`#!/usr/bin/env node`) makes it executable from the command line
- ES module syntax (`import`) for consistency with modern Node.js
- `program.version()` with `-v, --help` flags per Commander conventions
- Subcommands are stubs — `install/update/uninstall/doctor` print placeholder messages; actual implementation in later stories
- `program.parse()` consumes argv and routes to the appropriate command
- Help shown automatically if no arguments provided (UX convenience)

## Technical Details

### Stack constraints (from PRD and system/CLAUDE.md)

- **Node.js version:** ≥ 20.0.0 (hard requirement enforced in `package.json` `engines`)
- **Module system:** ES modules (type: "module") for modern async/await and native module support
- **Package dependencies:** Limited to `@clack/prompts`, `commander`, `picocolors`
  - `commander` — CLI argument parsing and subcommand routing (industry standard)
  - `@clack/prompts` — TUI prompts for the interactive wizard (added in later stories)
  - `picocolors` — terminal colors for the wizard UI (added in later stories)
- **Testing:** Node's native `node:test` module (built-in, no external test frameworks)

### Architecture notes

- **Entrypoint pattern:** The `bin/ralph.js` shebang and `package.json` `bin` field enable both direct execution (`node bin/ralph.js`) and npm-script-based invocation
- **Version sourcing:** Reading version from `package.json` ensures a single source of truth; avoids manual sync
- **Stub actions:** All subcommands print "not yet implemented" and exit cleanly — this allows the CLI skeleton to be tested before implementation
- **No external config:** The installer has no config files yet; all questions/answers come from the interactive wizard (Story 2.2)

### Edge cases and gotchas

1. **Windows line endings:** If `bin/ralph.js` is checked in with CRLF (`\r\n`), the shebang will fail on Unix systems. Ensure `.gitattributes` or `.editorconfig` normalizes to LF.
   - Workaround: Verify with `file bin/ralph.js` — should show "POSIX shell script" not "CRLF".

2. **Node.js version mismatch:** A user with Node 18 or 19 running `npm install` will be warned but may proceed, then hit "unsupported version" at runtime.
   - Mitigation: The CLI's first action (once implemented) will check `process.version` and refuse to run if < 20.

3. **Commander's default help:** If no command is provided, Commander shows help automatically. The extra check for `process.argv.length === 2` is redundant but harmless for UX consistency.
   - Acceptable: Leave it in; it makes the intent explicit.

4. **test/ directory discovery:** Node's `node:test` discovers files matching `'test/**/*.test.js'` by default. Ensure test files end in `.test.js`.
   - If a future story adds tests in a different pattern (e.g., `test/unit/cli.spec.js`), the glob needs updating.

5. **Empty directories and git:** Git doesn't track empty directories. If a future story needs to populate `src/`, `templates/`, or `scripts/`, ensure `.gitkeep` placeholders are added if needed.
   - Current approach: Leave them empty; they'll be ignored until they contain files.
