**Status:** ready for implementation

## Story statement

The installer must support fully automated (non-interactive) operation for scripting and CI/CD use cases. Users can run `npx <package> install --yes --force` into an empty directory with no prompts, no spinners, and plain output; individual flags allow per-question overrides; conflicts without `--force` fail fast non-zero. Output respects `NO_COLOR` and omits ANSI codes when stdout is not a TTY.

## Acceptance criteria

1. **Non-interactive with `--yes`:** Running `installer/bin/ralph.js install --yes --force` into an empty fixture directory completes without prompts and exits 0; the resulting install passes `doctor`.

2. **Plain output in pipes:** Redirecting install output to a file (e.g., `... install --yes --force > output.txt 2>&1`) yields zero ANSI escape codes; no spinners or color appear in the file.

3. **Conflict detection:** Running install a second time with conflicting state (e.g., existing `scripts/ralph-loop.sh`) without `--force` exits non-zero immediately, without prompting the user.

4. **NO_COLOR respected:** Running with `NO_COLOR=1` produces zero color codes even when stdout is a TTY.

5. **Per-question flags:** `--list-options` outputs all available flags; each wizard question has a corresponding command-line flag (e.g., `--app-dir`, `--checkpoint-command`); providing a flag skips the corresponding prompt.

6. **Non-TTY plain output:** Both spinners and color codes are stripped when `process.stdout.isTTY` is false, independent of `NO_COLOR`.

---

## Implementation overview

This story modifies the wizard and writer to accept command-line arguments and detect TTY/color state. It does **not** refactor the installer's architecture — the InstallPlan in-memory representation remains unchanged.

### Files to modify

1. **`installer/bin/ralph.js`** — Already exists (Story 1.2). Add global flags: `--yes`, `--force`, `--list-options`, and per-question flags.

2. **`installer/src/wizard.js`** — Already exists (Story 2.2). Modify to accept pre-filled answers from CLI args; skip prompts for questions answered via flags; use defaults when `--yes` is set.

3. **`installer/src/writer.js`** — Already exists (Story 2.3). Modify conflict detection to fail fast non-zero when conflicts exist and `--force` is not set.

4. **`installer/src/cli-parser.js`** — New file. Utility to parse per-question flags and validate them; exports a function that returns a map of question keys to CLI values.

5. **`installer/src/colors.js`** — New file. Utility for conditional color output: detects TTY, parses `NO_COLOR`, provides a single `color(text, code)` function that returns plain text when colors are disabled.

## Detailed implementation steps

### Step 1: Create `installer/src/colors.js` — TTY and color detection utility

This module centralizes the logic for detecting whether colors and spinners are safe to use.

**Exports:**
- `isColorEnabled()` → `boolean`: returns `false` if `NO_COLOR` is set OR `process.stdout.isTTY` is false.
- `colorize(text, picocolorsFunction)` → `string`: applies the color function if `isColorEnabled()`, otherwise returns plain `text`.
- `note` → `string`: a normalized marker (e.g., `→`) for use in both TTY and non-TTY output (avoids Unicode issues in pipes).

**Implementation note:** Check `process.env.NO_COLOR` (defined = disabled, value irrelevant per the [NO_COLOR spec](https://no-color.org/)); check `process.stdout.isTTY` to detect piped output.

**Test coverage:** Unit tests with stubbed `process.stdout.isTTY` and `process.env.NO_COLOR` to verify both conditions work independently and together.

### Step 2: Create `installer/src/cli-parser.js` — Per-question flag parsing

This module parses and validates CLI flags corresponding to each wizard question.

**Context:** The wizard flow (Story 2.2) has these questions:
- Target directory (defaulted from cwd)
- Target classification (auto-detected)
- App directory (e.g., `src`)
- Checkpoint command (e.g., `npm run build && npm test`)
- Stack description (free text)
- Use BMAD (yes/no)
- Task source: scaffold or point to existing PRD/epic (choice)
- Skip npm script (yes/no)

For each question, define a CLI flag:
- `--app-dir <path>` → app directory
- `--checkpoint-command <cmd>` → checkpoint command
- `--stack-description <text>` → stack description
- `--use-bmad <yes|no>` → use BMAD (boolean flag or choice)
- `--task-source <scaffold|existing>` → task source (scaffold commented stubs or point to existing)
- `--skip-npm-script <yes|no>` → skip adding npm script

**Exports:**
- `parseCliArgs(commander)` → `{key: value, ...}`: extracts values from the Commander program object; returns an object mapping question keys to CLI-provided values. Keys are omitted if the flag was not provided.
- `validateCliArgs(cliArgs)` → throws if a value is invalid (e.g., `--use-bmad foo` where foo is not `yes` or `no`).
- `listOptions()` → `string`: returns a formatted list of all available flags with defaults and descriptions (for `--list-options` output).

**Test coverage:** Unit tests for each flag, including validation (valid and invalid inputs).

### Step 3: Modify `installer/bin/ralph.js` — Add global flags to the install subcommand

Extend the Commander program with new global options:

```bash
-y, --yes                    Accept all defaults without prompting
-f, --force                  Skip conflict prompts and overwrite
--list-options              Print available flags and exit
--app-dir <path>            App directory (e.g., 'src')
--checkpoint-command <cmd>  Checkpoint command
--stack-description <text>  Stack description
--use-bmad <yes|no>        Install BMAD alongside the loop
--task-source <mode>       scaffold | existing
--skip-npm-script <yes|no> Skip adding npm script
```

**Logic flow:**
1. If `--list-options` is set, call `cli-parser.listOptions()`, print it, and exit 0.
2. If `--yes` is set, mark all questions as "use defaults" (passed to wizard).
3. If any per-question flag is set, validate it and pass to the wizard as a pre-filled value.
4. If `--force` is set, pass it to the writer so conflicts don't prompt.

### Step 4: Modify `installer/src/wizard.js` — Accept pre-filled values and skip prompts

The wizard function signature changes minimally:

```javascript
async function wizard(options = {}) {
  // options = {
  //   cliAnswers: { appDir, checkpointCommand, ... },  // from CLI
  //   useDefaults: true/false,  // from --yes
  //   isNonInteractive: true/false,  // derived from isColorEnabled()
  // }
  
  // If useDefaults is true, skip all prompts and use defaults
  // If cliAnswers has a key, use that value instead of prompting
  // Otherwise, prompt normally
}
```

**For each prompt:**
1. Check if the answer came from a CLI flag (`cliAnswers.appDir`, etc.); if yes, use it.
2. Check if `--yes` was set; if yes, use the default.
3. Otherwise, show the prompt normally.

**Output behavior:**
- When `process.stdout.isTTY` is false or `NO_COLOR` is set, suppress all color and spinner output from `@clack/prompts`. (This may require wrapping the prompts or using a non-TTY-aware prompt library — investigate `@clack/prompts` docs; it may already have a flag for this.)
- If suppressing is not automatic, log plain lines instead of using `@clack/prompts` for non-TTY runs.

**Test coverage:** Unit tests for each code path (CLI override, `--yes` default, normal prompt). Integration test: run with `--yes` and verify no prompts appear.

### Step 5: Modify `installer/src/writer.js` — Fast-fail on conflicts without `--force`

The writer's conflict-detection logic must change:

**Current behavior (from Story 2.3):** For each conflict, prompt the user.

**New behavior:**
1. Detect conflicts (files that would be overwritten).
2. If conflicts exist AND `--force` is not set:
   - Log a plain error message: `"Conflict: installer/bin/ralph.js already exists. Use --force to overwrite."`
   - Exit process with code 1 immediately. **Do not prompt.**
3. If `--force` is set, overwrite without prompting.

**Output:** The error message must be plain text (no colors, no spinners).

**Test coverage:** E2E test that runs install twice into the same directory without `--force`, verifying exit code 1 and no prompts.

### Step 6: Add integration test for non-interactive mode

Create or extend `installer/src/wizard.test.js` or `installer/test/` to include:

**E2E test: `--yes --force` into empty dir**
- Use a temporary fixture directory.
- Run `npx ralph install --yes --force --directory <fixture>`.
- Verify exit code 0.
- Verify all expected files exist (per FR-6: `scripts/ralph-loop.sh`, `.ralph/manifest.json`, etc.).
- Run `doctor` on the result and verify exit code 0.

**E2E test: Piped output has no ANSI codes**
- Run install with output redirected to a file.
- Read the file and verify no ANSI escape sequences (pattern: `\x1b\[` or `\033\[`).

**E2E test: Conflict without `--force` exits non-zero**
- Run install into the same fixture twice.
- Second run without `--force` must exit non-zero and not prompt.

**Unit test: `--list-options` output**
- Mock Commander and verify the flag is recognized.
- Verify the output lists at least 5 flags.

## Technical notes

### Non-TTY detection

```javascript
const isTTY = process.stdout.isTTY === true;
const noColor = process.env.NO_COLOR !== undefined;
const colorEnabled = isTTY && !noColor;
```

### Spinner suppression

`@clack/prompts` may have a built-in TTY check. If not, wrap all spinner calls in a check:

```javascript
if (colorEnabled) {
  spinner.start('Installing...');
  // ... work ...
  spinner.stop('Done');
} else {
  console.log('Installing...');
  // ... work ...
  console.log('Done');
}
```

### Conflict detection in writer.js

When iterating over files to write, collect conflicts first:

```javascript
const conflicts = files.filter(f => fs.existsSync(f.path) && !manifest?.owns(f));
if (conflicts.length > 0 && !options.force) {
  for (const file of conflicts) {
    console.error(`Conflict: ${file.path} already exists. Use --force to overwrite.`);
  }
  process.exit(1);
}
```

### Validation of per-question flags

Example for `--task-source`:

```javascript
if (cliArgs.taskSource && !['scaffold', 'existing'].includes(cliArgs.taskSource)) {
  throw new Error(`Invalid --task-source: ${cliArgs.taskSource}. Expected: scaffold | existing`);
}
```

## Edge cases and gotchas

1. **Prompt library non-TTY handling:** `@clack/prompts` may not automatically suppress spinners in non-TTY mode. If not, wrap all prompts in a TTY check or provide a no-op mock for non-TTY runs. Investigate the library's behavior before starting implementation.

2. **Unicode in non-TTY:** Some prompt libraries emit Unicode characters (arrows, checkmarks) that may not render correctly when piped. Use ASCII alternatives when `isTTY` is false.

3. **Spinner is async:** If spinners are created but the output is piped, ensure they don't hang or throw. Test thoroughly in a pipe context.

4. **Manifest ownership:** When deciding whether to prompt on a conflict, check the manifest's ownership class. Installer-owned files should be overwritable with `--force`; user-owned files should conflict and block unless overwritten.

5. **Order of checks:** TTY check should happen early so that color/spinner decisions propagate throughout the run. Don't check `isTTY` multiple times; cache it.

6. **Exit codes:** Be consistent:
   - Exit 0: success, no changes needed
   - Exit 1: conflict without `--force`, validation error, write failure
   - Exit 2: preflight failure (e.g., Node < 20)

7. **NO_COLOR spec:** The standard says the *existence* of the `NO_COLOR` env var disables colors, regardless of its value. Check `process.env.NO_COLOR !== undefined`, not the truthiness of its value.
