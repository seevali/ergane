### Architecture Context

The installer is a self-contained npm package living in `installer/`. Its published footprint includes:

- `bin/ralph.js` (entry point)
- `src/*.js` (all CLI modules)
- `templates/loop/` (synced copy of `scripts/ralph-loop.sh` and `scripts/prompts/`)
- `templates/stubs/` (PRD/epic templates)
- `README.md` (from `installer/README.md`)
- `LICENSE` (from repo root)

The package.json's `files` array controls what npm pack includes; `bin` entry points to `bin/ralph.js`.

### Step 5: Run `npm pack` dry-run and measure size (5 min)

**Instructions:**

1. **Run npm pack:**
   ```bash
   cd installer

   ```
   - This simulates packing without creating the tarball.
   - Output should list all files that will be included.

2. **Create the actual tarball:**
   ```bash
   npm pack
   ```
   - This creates `@seevali-ralph-loop-0.1.0.tgz` (or similar based on your chosen name).

3. **Measure unpacked size:**
   ```bash
   # Extract and measure
   mkdir -p /tmp/ralph-pack-test
   cd /tmp/ralph-pack-test
   tar xzf /path/to/installer/@seevali-ralph-loop-0.1.0.tgz
   du -sh package/  # Human-readable total size
   du -sh package/*  # Size per top-level directory
   ```

4. **Verify size constraint:**
   - Unpacked size must be **< 1 MB** (1024 KB).
   - Expected typical size: **400–600 KB** (bash scripts + Node modules are light; `@clack/prompts` is ~200 KB alone).
   - If size exceeds 1 MB:
     - Check what's bloating it: `du -sh package/node_modules/` often reveals unnecessary dependencies.
     - Review `files` array in `package.json` — is anything unnecessary included?
     - Consider whether a dependency can be removed (though all three listed dependencies are required).
     - If unavoidable, adjust AC3 budget upward with a note and escalate.

5. **Clean up:**
   ```bash
   cd installer
   rm -f @seevali-ralph-loop-0.1.0.tgz  # Remove the test tarball
   cd -
   rm -rf /tmp/ralph-pack-test
   ```

**Verification:**
```bash

# (Then verify unpacked size < 1 MB as shown above)
```

**Common issues:**

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| Size > 1 MB | Extra dependencies or templates | Check `node_modules/`, remove unused files from `templates/` |
| Pack fails (missing files) | `files` array incomplete or wrong path | Verify `bin/ralph.js`, `src/`, `templates/` all exist and are readable |
| Pack includes unwanted files | Missing `.npmignore` or `files` not restrictive | Use explicit `files` array (not `.npmignore`) to whitelist only needed paths |

## Edge Cases & Gotchas

### Edge Case 1: Chosen package name becomes unavailable

**Scenario:** You resolve the name as available in Step 1, but by the time you `npm publish` a human picks it up.

**Mitigation:** This is rare (npm doesn't auto-release reserved names). If it happens:
- Escalate with documentation of the timing
- Pivot to the second-choice name (e.g., if `@seevali/ralph-loop` is taken, use `ralph-loop-installer` instead)
- Re-run Steps 2–5 with the new name
- Update `PUBLISHING.md` with the new package name before human publish

### Edge Case 2: `npm pack` size creeps above 1 MB

**Scenario:** Some dependency's sub-dependency grows larger than expected.

**Investigation:**
```bash
npm ls --depth=3  # Show dependency tree with versions
du -sh installer/node_modules/@clack  # Verify @clack/prompts is not bloated
```

**Mitigation:**
- If a direct dependency (e.g., `@clack/prompts`) has grown: check if a minor version bump introduced larger dependencies. Evaluate pinning to an older version or finding a lighter alternative.
- If an indirect dependency is bloated: check the direct dependency's `package.json` for optional dependencies that can be excluded.
- If unavoidable: document in `PUBLISHING.md` the actual size and why it's acceptable (e.g., "0.1.0 is 1.1 MB due to X; acceptable for the feature value").

### Edge Case 3: License file not found

**Scenario:** Repo root has no `LICENSE` file.

**Fix:** Add MIT LICENSE at repo root (see Step 4). npm requires a license file for publication.

### Edge Case 4: 2FA blocks the publish step

**Scenario:** npm account has 2FA enabled (required for publish), but operator doesn't have authenticator configured.

**In `PUBLISHING.md`:** Document this clearly under "Publish workflow → Step 2" so the operator is not surprised.

**Fix:** Operator must generate a one-time password (OTP) from their authenticator app when prompted.

### Edge Case 5: Previous publish of same version exists

**Scenario:** You accidentally published 0.1.0 before, and npm prevents re-publishing the same version.

**Mitigation:** Versions are immutable on npm. If you need to re-publish:
- Bump the version to 0.1.1 in `package.json`
- Depracate 0.1.0 in `PUBLISHING.md` (see Rollback section)
- Publish 0.1.1

### Edge Case 6: Templates sync is out of date

**Scenario:** A subsequent change modifies `scripts/ralph-loop.sh` or `scripts/prompts/`, but `installer/scripts/sync-templates.sh --check` was not run.

**Prevention:** This story should not modify loop files. If it does (by mistake), the code reviewer will catch it with the hard block:

## Implementation Notes for Dev Agent

- **Do not** run `npm publish`. The checklist is the deliverable; actual publish is human-driven.
- **Do** verify every field in `package.json` against the metadata you set. npm is strict about JSON syntax.
- **Do** test the pack size twice (once with `--dry-run`, once with actual tarball).
- **Do** include clear, copy-paste instructions in `PUBLISHING.md`. The human operator is likely unfamiliar with npm tooling.
- **Do** ensure `PUBLISHING.md` is self-contained: a reader with no prior context can follow it start-to-finish and successfully publish.
- **Be careful** with the `repository.directory` field — if omitted, npm assumes the package is at repo root. We need `"directory": "installer"` to point to the subdirectory.
